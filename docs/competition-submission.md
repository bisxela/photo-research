# Photo Search 比赛提交说明

## 项目定位

Photo Search 是一个基于 Chinese CLIP 的图片语义检索 Demo，目标是把“上传图片、生成向量、文本搜图、以图搜图”整合成一个可以直接演示的 Web 工作台。

当前更适合比赛展示和小规模共享使用，而不是完整生产化产品。

当前版本已经支持基于本机后端的“正式账号雏形”。

## 当前已经实现的功能

### 1. 图片上传

- 支持单张上传
- 支持批量上传
- 支持常见图片格式校验
- 支持文件大小限制

### 2. 图片预处理

- 自动提取宽高、格式、文件大小
- 自动生成缩略图
- 自动保存原图与缩略图访问地址

### 3. 图片去重

- 后端基于 SHA-256 checksum 去重
- 重复上传同一文件时，直接复用已有图片记录

### 4. 向量化与检索

- 上传后异步生成 Chinese CLIP 图片向量
- 支持文本搜图
- 支持基于已有图片做相似图检索
- 支持搜索日志统计

### 5. 前端演示能力

- 单页式工作台
- 支持用户注册与登录
- 支持本地保存登录会话
- 可检查后端、数据库、模型状态
- 可轮询图片编码状态
- 可查看缩略图和原图
- 已提供 OCR 搜索入口
- 新上传图片可自动触发本地 OCR
- 可在图片详情中执行本地 OCR 文字识别
- OCR 结果可回写后端，并参与后续 OCR 搜索
- 支持移动端浏览和基础 PWA 安装

## 已验证的可演示能力

在本地实际联调中，以下流程已经跑通：

1. 启动 Docker 后端
2. 健康检查返回正常
3. 上传测试图片成功
4. 后台向量编码完成
5. 文本搜图返回结果

## 当前更适合怎样使用

### 已支持

- 个人本地使用
- 同一台主机上的多账号访问
- 局域网访问
- 通过 Tailscale 让少量朋友远程访问同一套服务

### 暂不支持

- 完整生产级账号体系
- 强安全权限隔离
- 每个用户单独的搜索记录治理
- 云端多租户部署

## 多人协同是否已经具备

结论：目前已经具备“基于账号雏形的多人协同”能力，但还不是完整生产级账户系统。

当前实现方式是：

- 后端新增 `users` 表
- 支持注册、登录和 `Bearer Token` 会话
- 后端图片、查询、轮询、相似图检索都按当前账号过滤
- 上传文件按用户目录保存
- 同一台主机可以服务多个不同账号

这意味着当前已经可以做到：

- 多个设备访问同一台本机服务
- 不同设备登录各自账号
- 文本检索只返回当前账号的图片
- 相似图检索只在当前账号内进行
- 同一账号在不同设备上访问同一图库

但当前仍然不能算完整产品化用户系统，因为还缺：

- 强安全权限控制
- 管理员能力
- 云端多租户部署

## 为什么这仍然还只是“账号雏形”

虽然现在已经有注册和登录，但它本质上仍然是 Demo 级方案：

- token 签名密钥仍是本地固定配置
- 没有密码找回、邮箱验证、验证码等能力
- 没有更细粒度的权限控制
- 没有生产级安全加固和审计
- 更适合比赛展示，不适合直接作为正式产品上线

这意味着现在可以做的是：

- 多个人共用同一套检索服务
- 多个人各自在自己的账号下上传和搜索
- 同一台主机上演示“不同账号对应不同图库”

但不能做的是：

- 生产级安全账号体系
- 完整可信的身份治理
- 面向公网长期暴露的正式服务

## 如果没有服务器，为什么不方便实现多人分别存储

如果没有一台持续在线的后端服务，多人独立存储会很不方便，原因很直接：

- 用户身份需要服务端保存和校验
- 图片文件需要统一存储位置
- 向量索引需要统一维护
- 跨设备访问需要共享数据库和文件系统
- 搜索结果必须基于“当前用户自己的图片集合”过滤

如果完全没有服务器，只靠前端网页：

- 无法可靠保存多人共享数据
- 无法统一做向量化和数据库检索
- 无法在不同设备之间同步私有图库

因此，多人协同至少需要一个中心化后端，哪怕只是你自己的单机服务。

## 访问方式是否必须依赖 Tailscale

不是必须。

当前项目可用的访问方式有：

- 同一局域网内直接访问主机 IP
- 同一热点或同网段设备访问
- 使用 Tailscale 做异地或更稳妥的私网访问
- 后续使用 tunnel / 反向代理做公网访问

如果只是比赛现场展示，多数情况下局域网访问已经足够，Tailscale 只是可选增强方案。

## 适合比赛时如何表述

建议把当前项目定义为：

- 一个可运行的图片语义检索 Demo
- 已完成从上传到检索的核心闭环
- 已支持个人使用和小范围共享访问
- 已支持基于账号的上传与检索隔离
- 下一阶段再向正式多用户独立图库演进

这样表述更稳妥，也符合现在代码的真实状态。

## 后续规划

### P1. 比赛提交前建议完成

- 准备一组更干净的演示图片集
- 补充 3 到 5 个固定演示查询词
- 录制或截图上传 -> 搜索 -> 相似图检索流程
- 在 README 中明确启动方式和项目亮点

### P2. 小规模共享版本

- 保持单机部署
- 用局域网或 Tailscale 方式共享访问
- 保持当前账号雏形方案
- 补一个“注册 / 登录 / 双设备演示”的说明页

### P3. 多用户版本

要支持“多人分别存储”，建议新增：

1. `users` 表
2. `images.user_id`
3. 登录鉴权机制
4. 上传路径按用户分目录或对象存储前缀隔离
5. 搜索 SQL 按用户过滤
6. 搜索日志绑定用户

### P4. 更适合长期部署的方向

- 把本地文件存储迁移为真正使用 MinIO 或云对象存储
- 用生产模式部署前端，而不是 `next dev`
- 为后端增加鉴权、限流、异常监控
- 为检索质量增加标签、重排或更好的评估集

## 当前结论

这个项目现在已经足够作为比赛作品提交，前提是你把它定位成：

- 语义图片检索 Demo
- 本地/单机可运行
- 支持共享访问
- 已具备账号级隔离雏形，但未完成生产级安全体系

如果你要把“正式可上线的多用户产品”作为卖点，那么当前代码还不够，还需要继续补安全、账号治理和部署体系。
